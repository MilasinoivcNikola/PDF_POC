## 2026-06-08 — Generation Progress & Orchestration API

**Branch:** `feature/generation-progress` → `main` (`b59011e`, merge `241f4a4`) · Craft Area 3 (nextjs-ui-builder)

Wired the wizard's "Generate" to the real illustration pipeline (feature 07) with an honest per-illustration progress screen, then auto-advance to preview: complete the wizard → click Generate → watch the backend build the book → land on `/create/preview`. Replaced the `/api/generate-illustrations` 501 stub and extended the Generate step; added the progress component, a poll hook, and a label module (+2 test files).

- **Transport decision — polling on disk state + an in-memory job registry (NOT SSE).** Documented in the route header. Generation runs minutes (gpt-image-2 ≈ 5 image-input req/min, ~4–8 min/book), feature 07's `generateAllIllustrations` writes each PNG to `./generated/[id]/` incrementally but exposes no per-image callback, so the most honest progress signal is the set of PNGs already on disk. Polling that survives a page refresh where a held-open stream would not; the single local process makes a module-level job registry sufficient (no queue).
- `app/api/generate-illustrations/route.ts` — `POST` reads the session, **claims the run synchronously** (check-and-set before the first `await`, released on write failure), flips `status:"generating"`, persists, then fires `generateAllIllustrations` **fire-and-forget** (returns `{ ok:true, status:"generating", total:14 }` at once). On resolve it persists the 14-entry manifest + `status:"ready"`; on reject the `.catch` stashes the message on the in-memory job (no unhandled rejection). `GET ?id=` derives progress from `./generated/[id]/*.png` (traversal-guarded via `resolveUnder`), returns `{ ok, status, done, total:14, donePages, error? }` with a status-inference fallback (live job → `session.status` → disk-complete) so a server-restart/cache-hit run still resolves. House JSON shape throughout.
- `components/wizard/GenerationProgress.tsx` + `useGenerationProgress.ts` — the `generating.html` screen (rotating ornament, "NN of 14", animated bar, 14-item done/current/pending checklist, pet-description note, cost/quality footer) driven by a poll hook (POST-to-start, poll ~1.8s, single loop guaranteed by a `stopRef`, cleanup on unmount). Auto-pushes `/create/preview` on `ready`; an errored run shows a gentle notice + "Try painting again" retry.
- `components/wizard/illustrationLabels.ts` — `ILLUSTRATION_SLOTS = ["reference", ...SCENE_PAGE_IDS]` (**14**, driven from the real pipeline — not the prototype's hand-picked 12) + warm pet-name-interpolated labels.
- `app/create/generate/page.tsx` — after the `/api/session` write, hands into `<GenerationProgress>`. **Refresh-resume:** on mount it asks `GET /api/generate-illustrations?id={draft.id}` and, instead of re-showing the Generate CTA, re-enters the live progress screen if a run is `generating` (the POST it fires is an idempotent no-op for an in-flight run) or redirects to preview if already `ready` — with a brief "Gathering your story…" gate to avoid a flash.

**Review (PASS after one fix).** Code review found a blocking **TOCTOU race**: the idempotency guard read the job map *before* the job was registered (inside `startGeneration`, after `await writeSession`), so two interleaved POSTs — reachable via React Strict Mode's double-mount, a double-click, or a duplicate tab — could launch two concurrent **paid** runs. Fixed by claiming the job synchronously before the first await; added an interleaved double-POST regression test (fails pre-fix with 2 runs, passes post-fix with 1).

**Live QA (PASS).** All five acceptance checks + both edge cases held: real watercolor PNGs of the actual dog, live checklist read "10 OF 14", auto-advance to preview with a 14-entry manifest on disk, footer copy, and — critically — a **mid-run browser refresh resumed from disk without restarting** and **10 POSTs produced exactly 1 generation run** (idempotency holds live). The resume UX fix above was then verified for **$0** via curl contract checks (ready→preview, generating→progress at 13/14, missing→CTA). **Cost lesson:** this QA ran at Medium and, across happy-path + refresh + idempotency testing, cost ~$3 — it didn't need to. Locked in a standing rule (qa-verifier "Cost discipline" + `qa` action guardrail + memory): QA flow/orchestration features by **reusing an existing `./sessions/` fixture** (re-running an identical session is a free cache hit), never fresh Medium books.
