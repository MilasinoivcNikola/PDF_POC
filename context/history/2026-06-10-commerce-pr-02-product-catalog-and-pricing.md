## 2026-06-10 — Commerce PR-02: Product Catalog & Pricing

**Branch:** `feature/product-catalog` → `main` (`337ccf0`, merge `d595580`) · Commerce Foundation (nextjs-ui-builder)

Commerce build, PR-02 — the **catalog layer**: turns the `StoryDefinition` registry into a sellable catalog so the future storefront (PR-04) and checkout (PR-06) have products to list and charge for. A small, pure, **client-safe** data module — a sibling of the PR-01 order model, no UI. Imports the catalog contract every later commerce PR builds on.

- `lib/catalog/products.ts` (new) — the `Product` storefront/checkout contract (`productId`, `storyType`, `title`, `tagline`, `description`, `priceUsd` [US cents], `lsVariantId?`, `sampleImages`, `illustrationCount`) + `getProducts()` / `getProduct(productId)`. One `Product` per live `storyType` (`story-1-book`, `story-2-letter`). Field naming/JSDoc match the `lib/order/types.ts` style.
- **No-drift guard (the load-bearing invariant):** `illustrationCount` is `getStory(storyType).illustrationSlots.length` — read from the registry, never hardcoded (resolves to 13 for Story 1, 2 for Story 2). The catalog **references** the registry, never forks it — so a future Story 3 registered in `lib/story/registry.ts` is the seam for a new catalog entry, exactly as the commerce roadmap describes ("each new title = a new registry entry that auto-appears as a purchasable product").
- **Client-safe is the headline constraint.** The public storefront imports this module, so its graph must never transitively pull Puppeteer or any `node:`/fs engine code (a stray engine import breaks the storefront's static build). It imports only `getStory` + the `StoryType` type; the registry graph was made puppeteer-free in PR-19 (filename builders split into `lib/pdf/filename.ts`). **Verified:** a full import-closure trace of `products.ts` (17 modules) found zero puppeteer/`node:`/fs/`react-dom/server`/openai imports, `grep puppeteer .next/static` → 0, and all `/create/*` routes still prerender static `○`.
- **Pricing is config, not content.** Both products ship `priceUsd: 2900` ($29) as explicitly-labelled `PLACEHOLDER_*` constants, flagged as the deferred PM decision set before PR-06; `lsVariantId` left `undefined` until the Lemon Squeezy products exist. Honors the roadmap's "pricing deferred, needed by Phase 2/PR-06" rather than locking a number. No new dependency (`package.json` untouched); pure functions, no IO.
- `getProducts()` memoizes the catalog in a module-level cache (`catalogCache`) — lazy build on first call, stable reference across calls (a test pins this). Sound for read-only data; the only latent footgun is that immutability isn't type-enforced (`readonly`/`Object.freeze`) — non-blocking, no consumers yet.
- **Doc drift resolved in the same PR** (context-audit gate): added a "Commerce catalog" structural pointer to `coding-standards.md` naming `lib/catalog/products.ts` as the third commerce-layer sibling (after `lib/order/` + `lib/supabase/`) — pure/client-safe, the opposite discipline to the server-only Supabase client — so the commerce-layer map stays honest.

**Review (PASS) + context audit (DRIFT FOUND → resolved on-branch).** Code review found no blockers — verified 13/13 catalog tests, `tsc --noEmit` clean under strict, and the puppeteer-free import closure. Three non-blocking nice-to-haves: catalog immutability not type-enforced (cache returns shared `Product` references); `priceUsd > 0` is a comment-only invariant on a plain `number` (more relevant when real prices land in PR-06); the cold `catalogCache === null` build path is covered only incidentally. Context auditor flagged one staleness item — the PR-01 commerce-data-layer note in `coding-standards.md` didn't yet name the new `lib/catalog/` sibling — fixed on-branch per the same-PR discipline.
