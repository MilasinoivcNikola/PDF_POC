## 2026-06-10 — Commerce PR-01: Order Model & Supabase Data Layer

**Branch:** `feature/order-model-supabase` → `main` (`a6a9969`, merge `f168657`) · Commerce Foundation (nextjs-ui-builder)

Commerce build, PR-01 — **the data spine**, the keystone the whole commerce roadmap hangs off. Stood up an `Order` with a 10-state status machine, persisted in Supabase (Postgres + Storage) behind a server-only data-access layer. **No UI** — commerce PRs 02–10 import this contract. First infrastructure dependency since scaffolding (`@supabase/supabase-js`), pre-approved by the commerce roadmap. This is the first PR to graduate the project past the POC's "no database / no deployment" lines.

- `lib/order/types.ts` — the shared commerce contract every later PR imports: `Order` (`id`, `productId`, `storyType`, `status`, `customerEmail`, `inputs` = the captured `StorySession | Story2Session` **verbatim** so the engine consumes it unchanged, `photoKey`, `pdfKey?`, `lsOrderId?`, `deliveryToken?`, `error?`, `createdAt`, `updatedAt`) + the 10-value `OrderStatus` string-literal union.
- `lib/order/state.ts` — pure, IO-free state machine: `ALLOWED_TRANSITIONS` + `canTransition`/`nextStatus`/`isTerminal`/`assertTransition` (+ `IllegalTransitionError`). The **single source of truth** for legal moves. Spend guard (security-critical): the only edge into `generating` is `queued→generating` and into `queued` is `paid`/`failed` — **generation can never start on an unpaid order**. No-op self-transitions rejected; `delivered`/`refunded`/`cancelled` terminal; `failed` recoverable via `failed→queued`.
- `lib/order/store.ts` — `createOrder` (seeds `pending_payment`) / `getOrder` (`null` on not-found) / `updateOrderStatus` (calls `assertTransition` **before any DB write** — an illegal move throws and never touches Postgres) / `listOrdersByStatus`, plus the pure exported `rowToOrder`/`orderToRow` snake↔camel mapping (`null`↔`undefined` optionals, `inputs` JSON intact).
- `lib/supabase/server.ts` — server-only service-role client (lazy singleton), guarded by the `lib/session/disk.ts` `node:`-import discipline so it can never reach a client/public bundle; key read from env, never logged/returned. `lib/supabase/ids.ts` — `isSafeOrderId` (mirrors `lib/ai/paths.ts`'s `isSafeSessionId`). `lib/supabase/storage.ts` — `putPhoto`/`getPhoto`/`putPdf`/`signedPdfUrl` against two private buckets, id-guarded before any key build / client call.
- `supabase/migrations/0001_orders.sql` — `orders` table (10-state `CHECK` mirroring `state.ts`), indexes (status for `listOrdersByStatus`, unique partial on `ls_order_id`/`delivery_token`), an `updated_at` trigger, **forced RLS** with zero anon/authenticated policies (default-deny; service-role bypasses), and two **private** buckets `order-photos` / `order-pdfs`.
- `.env.local.example` — `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` (server-only), `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` (kept `OPENAI_API_KEY`).

**Decisions:** Order `id` is `text` (not `uuid`) to match the app's `createSessionId()` UUID-string shape end-to-end (avoids a cast layer; DB keeps a `gen_random_uuid()::text` default as belt-and-braces). `isSafeOrderId` is a deliberate **copy** of the `isSafeSessionId` regex rather than importing across from `lib/ai/`, so the commerce data layer doesn't depend on the AI craft area — trivially consolidatable later. Added only `@supabase/supabase-js` (runtime client), **not** the `supabase` CLI (a dev tool; migrations run via dashboard/`psql`). Non-blocking nice-to-have left for PR-09's author: `getOrder` throws on a malformed id rather than returning `null` (fails loud; a delivery-lookup caller may want 404-vs-500 separation).
