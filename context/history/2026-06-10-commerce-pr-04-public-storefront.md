## 2026-06-10 — Commerce PR-04: Public Storefront

**Branch:** `feature/storefront` → `main` (`8fa8fe2`, merge `eca20ba`) · Phase 1 (nextjs-ui-builder)

Commerce build, PR-04 — **Phase 1, the first customer-facing surface.** The always-on, Vercel-deployable marketing + book-catalog site: a reframed landing, a `/books` grid + `/books/[productId]` detail driven by the PR-02 catalog, an "Order this book" CTA pointing at a stubbed order route, and labelled policy placeholders. No payment, no engine — pure groundwork the order chain (PR-05→09) builds on. Static/SSG throughout; the load-bearing property is that the public route graph stays provably engine-free.

- **Storefront pages (all `app/(public)/`, server components, prerendered):** `books/page.tsx` (+CSS) — product grid from `getProducts()` (card per product: sample thumbnail, title, tagline, `$29`, link to detail), `○` static. `books/[productId]/page.tsx` (+CSS) — detail with a sample-art gallery, full description, price, registry-derived **illustration count** (13 for Story 1, 2 for Story 2 — proves it's not hardcoded), and the primary **"Order this book"** CTA; `generateStaticParams()` from the catalog + `notFound()` on unknown id → `●` SSG (both ids). `order/[productId]/page.tsx` (+CSS) — a gentle, **inert** "ordering opens soon" stub (no form/input/upload/payment — that's PR-05), `●` SSG. `policies/page.tsx` (+CSS) — refund / "How it's made + AI-honesty" / privacy as clearly-labelled **placeholders** with visible PM-signoff markers, `○` static.
- **Landing reframe (`app/(public)/page.tsx`):** hero kept; new primary CTA **"See the keepsakes" → /books**; the two product cards now link to their detail pages instead of seeding a local wizard draft. **Intentional behavior change:** the public landing no longer links to the operator `/create` wizard — correct for the commerce model (customer path = storefront → order; the operator reaches `/create` by URL). `StoryStartButton.tsx` left in the tree for the operator wizard.
- **Catalog + price (`lib/catalog/`):** new pure, client-safe `price.ts` (`formatPriceUsd(cents)` — `2900 → "$29"`, `2950 → "$29.50"`, clamps negatives/rounds fractional cents; 5 unit tests). `products.ts` `sampleImages` populated with the committed `public/samples/<productId>/` paths.
- **Sample art:** 5 real Low-tier watercolor pages pulled from existing `./generated/` books (canonical `b41b8df0` left untouched) — Story-1 cover + page-4 + page-10 (`feat13-low-otis`), Story-2 letter-cover + belief wash (`dc34eb11…`). Optimized with the built-in macOS `sips` (`-Z 800`, JPEG q78), **832 KB total, no new dependency.**
- **The one change beyond the literal page scope (behavior-preserving, flagged + reviewed):** adding `/books` to the boundary guard's `PUBLIC_ENTRIES` surfaced that the public closure reached `lib/ai/prompts` via `catalog → registry → story-1` (for `SCENE_PAGE_IDS`) — a path the guard bans by namespace even though `prompts.ts` is provably pure. Rather than weaken the guard, extracted the constant + type into a new neutral **`lib/story/scenes.ts`** (`SCENE_PAGE_IDS` / `SceneId`); `lib/ai/prompts.ts` **re-exports** them (every existing `@/lib/ai/prompts` consumer byte-for-byte unchanged); `lib/story/story-1.ts` imports from the neutral module. Net: the public closure reaches `lib/story/scenes.ts`, never `lib/ai/*`, so the guard keeps banning **all of `lib/ai/*`** from public outright (the stronger invariant). Same ids, same order → Story-1/Story-2 output and the registry's `illustrationCount` unaffected.
- **Boundary test (`lib/runtime/surface.boundary.test.ts`):** `PUBLIC_ENTRIES` += the four new public routes; the "non-trivial closure" sanity anchors re-pointed (the reframe dropped `StoryStartButton`/`lib/session/storage` from the public closure → now anchored on `app/(public)/books/page.tsx`, `lib/catalog/products.ts`, `lib/story/registry.ts`). Re-verified non-vacuous (injecting `import "@/lib/pdf/render"` into `/books` fails both engine-module + engine-package checks; reverted).
