## 2026-06-11 — Commerce PR-07: Local Batch Worker

**Branch:** `feature/order-worker` → `main` (`b6e3fb1`, merge `6e8cf14`) · Phase 3a (nextjs-ui-builder)

Commerce build, PR-07 — **Phase 3a, the automation moat.** A local, operator-only CLI (`npm run process:orders`) that drains `queued` Supabase orders, generates each book with the existing engine at **Low** tier, and parks each at `awaiting_review` (success) or `failed` (error) — **no manual generation step**. This is where automated-async replaces per-order labour. The worker *reuses* the engine without modifying `lib/ai/`; the only commerce-surface change is the new orchestration module + the CLI. The keystone property held throughout: **generation only ever runs for an order successfully claimed `queued → generating`** (the spend guard) and the engine/key path **never reaches the public deploy**.

- `lib/order/worker.ts` (new) — the orchestration logic with **injectable dependencies** (`WorkerDeps` / `defaultWorkerDeps`) so the whole flow is $0-unit-testable (store/Storage/engine/disk all mockable at the boundary). Exports `processQueuedOrders`, `processOrder`, the pure `detectImageExtension` / `scratchPhotoRelPath` / `buildEngineSession`, and `writePhotoToScratch`. Per order: **atomic claim** `updateOrderStatus(id,"generating")` **before any heavy await** (the feature-09 TOCTOU lesson) — an `IllegalTransitionError` on the claim → **skipped** (not failed, no generation, no spend); a generic claim error → re-thrown → counted failed but the order is **left `queued`** for the next run's natural retry. After the claim: download the photo (`getPhoto`) → local scratch `./uploads/[id]/photo.<ext>` → `generateAllIllustrations` (registry-driven slot list: Story-1 = 14, Story-2 = 2) → persist session + manifest to `./sessions/[id].json` (status `ready`) for the PR-08 admin → `awaiting_review`. On any post-claim error → `failed` with only `err.message` stored (never a full error object that could echo a key). Each order isolated in try/catch so one failure never crashes the batch (`failed → queued` stays recoverable). `ORDER_CONCURRENCY = 1` via the reused `mapWithConcurrency` — a single book already saturates the ~5 img/min ceiling through the engine's own pool, so serialising at the order level keeps the batch under the rate limit *and* serialises in-process claims.
- `scripts/process-orders.ts` (new) + `package.json` `process:orders` — thin CLI wrapper (mirrors `scripts/render-test.ts` + `scripts/tsconfig.json`), prints a non-secret succeeded/failed/skipped summary, exits non-zero if any order failed.
- **The photo-reconciliation integration detail:** the order's `inputs.pet.photo` (captured at intake, PR-05) is NOT a usable local path — intake stored the photo in Supabase and kept only the original file name. The worker downloads via `getPhoto(orderId)` and rewrites `session.pet.photo` to the path the engine resolves under `./uploads`. **`detectImageExtension`** sniffs the real format from the downloaded bytes (JPEG `FF D8 FF`, PNG `89 50 4E 47…`, WebP = `RIFF`@0-3 **and** `WEBP`@8-11 — the dual check rejects `RIFF…WAVE` audio; unknown/short → `jpg` fallback) and threads that one `ext` to BOTH the disk write and the engine path so they can't disagree — closing a **review-caught bug** where a small PNG/WebP (which `downscaleImage` passes through un-re-encoded under the ~1024px cap) was being written to `photo.jpg` and shipped to OpenAI mislabeled `image/jpeg`.
- No engine/store/state-machine changes; no new dependency. Reuses `mapWithConcurrency`/`withRetry`, the order state machine + `assertTransition` spend guard, `getPhoto`, the traversal guards, and `updateOrderStatus({ error })` (the `error` setter already existed).

**Reviews (code PASS · commerce-security PASS · context DRIFT FOUND → resolved on-branch).** Code review: one **blocking** finding — the hardcoded `.jpg` extension would mislabel a customer's small PNG/WebP photo to OpenAI — fixed with the byte-sniffing `detectImageExtension` (above) + tests; one accepted non-blocking note (a `failed → queued` retry re-spends the full book, because the manifest lives only in `./sessions/[id].json`, not back in `order.inputs` — low-frequency operator action, comments made honest, a natural PR-08 follow-up). Commerce security: **PASS** ("ship it") — spend guard holds, operator-only boundary intact (`grep` confirms nothing under `app/` imports the worker), IDOR contained (the order id is guarded at 3 points before any path/key build), no secret/PII in logs or the stored error. The **two-OS-process claim race** (`updateOrderStatus` is `UPDATE … WHERE id=?`, an app-level `assertTransition` not a DB compare-and-swap) was judged an **accepted residual** for a once/twice-daily operator batch command — documented in code + flagged for the operator; it flips to blocking only if generation ever becomes concurrent/daemon-triggered (the fix would be a conditional `.eq("status","queued")` store change, out of this PR's scope). Context audit: two doc drifts fixed on-branch — `coding-standards.md` now records `worker.ts` as the first engine-touching, operator-only `lib/order/` member; `commerce-roadmap.md`'s "drain **paid** orders" reconciled to "drain **queued**" (matching its own state machine — `paid → queued`) in both the architecture diagram (width-preserved) and Phase 3 prose.

**Live QA — the point of the feature (PASS 4/4, ~$0.024).** Driven via Bash against the **local `supabase start` stack** (`SUPABASE_URL`/service-role overridden inline; cloud project never touched — isolation proven, only `127.0.0.1` hit), real OpenAI key (re-verified HTTP 200). The **first** QA pass **caught a deterministic, every-order bug that 823 green unit tests + all three PASS reviews missed**: `scratchPhotoRelPath` returned `"<id>/photo.<ext>"` *without* the `uploads/` prefix the engine's `resolveUnder(cwd,"uploads",pet.photo)` requires, so every drained order failed at the photo guard *before* generating — and the unit test had **baked in the buggy value**, so the gates were green while the integration was broken (exactly why PR-07 earns a live run). Fixed with a one-line prefix (`uploads/${orderId}/photo.${ext}`) **plus a regression test that cross-checks the worker's output against the real engine guard** (`resolveUnder(cwd,"uploads",pet.photo)` must be non-null) so the two contracts can't silently drift again. Re-run → all four PASS: (1) seeded `queued` Story-2 order → `awaiting_review` with photo at `./uploads/[id]/photo.jpg`, 2 PNGs in `./generated/[id]/`, session `ready`/manifest len 2 on disk (the ~$0.012 spend, the headline that failed before); (2) **$0 idempotent re-run** — proven with a bogus key (no 401 = zero OpenAI calls), row + PNG md5s byte-identical; (3) injected failure (photo object missing) → `failed` with a secret-free message, $0, batch did not crash, good order still completed; (4) extension correctness. Total spend ~$0.024 (two Story-2 Low generations); canonical `b41b8df0` fixture intact.
