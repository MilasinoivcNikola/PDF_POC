## 2026-06-08 — Inline Preview Text Editing

**Branch:** `feature/preview-text-edit` → `main` (`890d064`, merge `6ce80d7`) · Craft Area 3 (nextjs-ui-builder)

Milestone 6, polish item 02. Let a parent correct **their own free-text inputs** (and the two names) directly on the in-browser preview, per page, beside the existing Regenerate control — so a typo or a reworded memory is a one-click fix on the preview screen instead of a wizard restart. **Option A** (edit the input → re-merge → re-render), chosen over Option B (free-form WYSIWYG override of any sentence): A keeps every edit inside the merge pipeline, so the hard "died"/no-euphemism rule and the age/death/belief variant logic stay enforced, and screen↔PDF parity is automatic (both still render `resolveStory(session)`). B (editing the fixed template prose) is deferred.

- `lib/story/editable-fields.ts` (new, pure + tested) — the **page→field map** (`petName`/`childName`/`parentDedication` on the cover-dedication spread; `breedColor` p2; `favoriteActivity` p4; `sleepingSpot` p5; `favoriteMemory` p6 — names edited **once** on the cover and propagated book-wide), the required-editable set, and helpers `setSessionField` (returns a NEW session with the cleaned value in the right nested group), `isEditableField`/`isRequiredField`/`isBlankAfterClean`, `getSessionFieldValue`. Reuses `clean()` from `merge.ts` (promoted from module-private to an export — one normalization, no duplication) so an edit gets the same trim / collapse-whitespace / strip-`{}`-braces treatment the wizard inputs get.
- `app/api/update-text/route.ts` (new) — `POST { id, field, value }`: traversal-guard the id (`isSafeSessionId`), validate the field against the `EDITABLE_FIELDS` allowlist (no arbitrary session-field writes), reject blanking a required field (`field_required`, no read/write), `readSession`, `setSessionField`, **re-run `resolveStory` BEFORE `writeSession`** (a `MergeError` → `story_incomplete` 422 with the on-disk session untouched — a bad edit can never corrupt the saved order), then persist and return the **full re-resolved `pages`** so the client swaps the whole book and global name edits propagate. House JSON shape throughout; near-clone of `regenerate-illustration/route.ts` minus the AI call (so a text edit is **$0**, no model call).
- `app/api/preview/route.ts` — additively returns the raw editable `fields` (the seven free-text/name values, all already present in the resolved `pages`) so the inline editors pre-fill. No new data exposed (no photo path, toggles, or secrets).
- `components/preview/{PageView,BookPreview}.tsx` — an **"Edit the words"** affordance beside Regenerate (shared `.preview-page__controls` row), opening an inline labelled `<textarea>` per page field with gentle bereaved-parent copy; on save → POST → swap the returned `pages`/header-names/`fields` into state, clear the now-stale `downloadMeta`. Required-blank is blocked client-side too; the Download guard is extended to `downloading || regenerating || savingText` so a mid-save render can't capture a stale book. Screen-only chrome around the **shared** `renderPage` — no page-markup fork.
- `app/globals.css` — generalized `.preview-page__regen` → `.preview-page__controls`/`__control` and added `.preview-page__editor*` (token-based, matching the `.field`/`.notice` patterns).
- 25 unit tests (`editable-fields.test.ts` + `update-text/route.test.ts`): the page→field map, `setSessionField` nested-write + `clean()`, unknown-field rejection (e.g. `species` rejected with no disk touch), required-blank detection, `parentDedication` clearing; route happy-path (returns re-resolved pages), required-blank rejection (no write), invalid field, invalid/missing id (traversal), 404, invalid JSON — disk layer mocked like the regenerate route's test.

**Correction to the spec wording (review nit):** the feature notes claimed `clean()` prevents a "double-article glitch" — it does **not**; `clean()` only strips braces and collapses whitespace, so a `breedColor` typed with a leading article ("a sweet mutt") still renders "was a a sweet…". That's a **pre-existing data property** (identical on screen and PDF, same as raw wizard input — the documented `b41b8df0` note), not introduced here; the new editor simply gives the user an easy path to fix it themselves. A real prevention (leading-article trim in the template or input) is a separate, out-of-scope concern. Deferred: Option B (free-form override editing) and making the enum/toggle answers editable.
