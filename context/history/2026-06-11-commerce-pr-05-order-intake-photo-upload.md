## 2026-06-11 — Commerce PR-05: Order Intake + Photo Upload

**Branch:** `feature/order-intake` → `main` (`67aa6bb`, merge `6482f3e`) · Phase 2a (nextjs-ui-builder)

Commerce build, PR-05 — **Phase 2a, the first customer write.** Turned PR-04's inert "ordering opens soon" stub into a real public order form: a customer provides pet/letter details + photo + email and creates a **`pending_payment`** `Order` in Supabase — **reusing the existing wizard inputs/validation, but with NO generation and NO charge.** Isolates intake + photo upload from the payment integration (PR-06 wires Lemon Squeezy). The keystone property: this is the **first `app/(public)/api/*` route**, so it defines the public *API* surface — the public host may write orders to Supabase, but still must never touch the engine.

- `app/(public)/api/order/route.ts` (new — the first public API route) — multipart `POST`: validates inputs/email/photo **server-side**, uploads the photo to Supabase **`order-photos`** (not the engine's local `./uploads/`), creates the order in `pending_payment`, returns `{ ok, orderId }` (house JSON shape). **No `assertOperator()`** — it runs on the public surface. Reuses the feature-08 required-field validators (`missingRequiredFieldsForDraft` + a `draftToSessionForDraft` backstop) so a later *paid* order can never `MergeError`; re-checks photo type/size with the operator upload route's constants (`ALLOWED_TYPES`, 10 MB cap); a `product_mismatch` guard rejects a story-2 product carrying story-1 inputs.
- `app/(public)/order/[productId]/{OrderForm,layout,page}.tsx` — the PR-04 stub becomes the real form. `OrderForm` (client) is product-aware (Story-1 child+memories or Story-2 owner+letter fields, shared pet fields, **email**, deferred photo upload, required-field gate) → on submit assembles via `draftToSessionForDraft` and POSTs to `/api/order` → confirmation stub. `layout.tsx` mounts `WizardProvider` (mirrors the operator `create/layout.tsx`). Page stays SSG (`●`) for both products.
- `lib/order/email.ts` (+test) — pure `isValidEmail`, single-sourced into the client gate and the server validation.
- **`createOrder` now accepts an optional caller-minted `id`** (`lib/order/{types,store}.ts`) — backward-compatible (absent → fresh UUID). The route mints the id once, `putPhoto(id, …)`, then `createOrder({ id, photoKey: photoKeyFor(id), … })`, so `photoKey === photoKeyFor(orderId)` is written **atomically** (no blank-then-patch). Order id is **server-minted, never client-supplied** (IDOR-safe).
- `components/wizard/ImageUploader.tsx` — additive optional `onUpload` prop defers the upload to submit (downscales via `downscaleImage`, hands the File back) instead of POSTing to the operator `/api/upload`; **operator-wizard behavior byte-identical** when omitted. `WizardProvider` gains `replaceDraft` to seed the chosen product into the mounted provider.
- **Boundary guard is now two-tier** (`lib/runtime/surface.boundary.test.ts`): public **pages** (`PUBLIC_ENTRIES`) stay fully client-safe (no engine **and** no `lib/supabase/server`); public **API routes** (`PUBLIC_API_ENTRIES`) may import the service-role Supabase client + `@supabase/supabase-js` but still must be **engine-free** (no `lib/ai/*` / Puppeteer / local-disk session IO). Proven non-vacuous (injecting `@/lib/pdf/render` into the route fails both engine assertions; reverted). The order *page* graph stays engine-free (it only `fetch`es the route).
