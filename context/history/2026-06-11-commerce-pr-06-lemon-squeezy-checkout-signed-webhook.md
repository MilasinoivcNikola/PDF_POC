## 2026-06-11 — Commerce PR-06: Lemon Squeezy Checkout + Signed Webhook

**Branch:** `feature/lemonsqueezy-checkout` → `main` (`93c87a2`, merge `9224d7a`) · Phase 2b (nextjs-ui-builder)

Commerce build, PR-06 — **Phase 2b, the money step.** Take payment via **Lemon Squeezy** (Merchant of Record) and advance a paid order through the state machine. The keystone security property: **payment is trusted ONLY via the signed, idempotent webhook — never the client redirect.** This PR drives `pending_payment → paid → queued` and **stops at `queued`** (PR-07's worker generates); there is no path that spends on an unpaid order. The webhook + checkout routes are the **second and third** secret-holding **public** API routes (after PR-05's `/api/order`), engine-free by the boundary guard.

- `lib/order/lemonsqueezy.ts` (new) — **pure, IO-free** core (the unit-test seam): `verifyWebhookSignature` (HMAC-SHA256 **hex over the raw body**, `crypto.timingSafeEqual` with a length guard so a wrong-length/non-hex signature returns false instead of throwing, missing/empty → false), payload parsers (`getEventName`/`getCustomOrderId`/`getLemonSqueezyOrderId`/`isPaidOrderEvent`), and a pure checkout-body builder + URL extractor. The only network IO — `createCheckout` (`POST /v1/checkouts`) — is a thin server-only fn kept out of the pure module. No new dependency (`node:crypto`). **LS API/signature/payload shapes confirmed via live docs** (context7 MCP was unavailable; WebFetch/WebSearch on docs.lemonsqueezy.com): `X-Signature` header, HMAC-SHA256 hex of raw body, `meta.event_name` (`order_created`/`order_paid`), `meta.custom_data.orderId` (in `meta`, not `data`), `data.id` = LS order id.
- `app/(public)/api/webhooks/lemonsqueezy/route.ts` (new, public — **no `assertOperator()`**) — the only trusted "paid" signal. **Order of operations is load-bearing:** read the **raw body** with `request.text()` BEFORE any parse (re-serializing would break the HMAC) → verify the signature → reject unsigned/tampered/mismatched **before any DB read/write** (401) → only then parse + act. Resolves our order via `meta.custom_data.orderId` (guarded by `isSafeOrderId`; `getOrder`'s throw-on-malformed-id is caught → clean 200, no 500 leak). **Idempotent:** reads the order first and advances step-by-step from its current status via `updateOrderStatus` (which `assertTransition`s before every write) — a fresh `pending_payment` moves once to `queued`, a partial-failure resume from `paid` runs only the remaining step, and anything already `queued`/mid-pipeline/terminal is a **no-op 200** (never an `IllegalTransitionError`). Unknown events → ignored 200. Persists `lsOrderId`, but **skips the write when LS sends no `data.id`** so a paid order never persists a blank `ls_order_id` (the post-review fix — stores SQL NULL, not `""`).
- `app/(public)/api/checkout/route.ts` (new, public) — server-side LS checkout creation: validates `orderId` (`isSafeOrderId`) + `productId` (catalog), resolves the LS **variant id from per-product env** (`LEMONSQUEEZY_VARIANT_<PRODUCT_ID>`), creates the hosted checkout with `checkout_data.custom: { orderId }` (round-trips into the webhook's `meta.custom_data`) + an origin-derived `redirect_url` → confirmation page, returns `{ ok, url }`. Missing config → `checkout_unavailable` 503; LS failure → `checkout_failed` 502; **neither leaks the env var name/value or the upstream error**.
- `app/(public)/order/[productId]/OrderForm.tsx` — after the unchanged PR-05 `/api/order` create, `POST /api/checkout` then `window.location` to the hosted URL; **graceful fallback** to the prior "we'll email a checkout link" confirmation if checkout isn't configured (so the form survives the not-yet-provisioned-LS state). Download/submit guards intact (no double-submit / redirect race).
- `app/(public)/order/[productId]/confirmation/page.tsx` (new) — SSG, informational-only "we've got it, check your email" page; **grants nothing, advances no state, trusts no query param** (the redirect is not proof of payment — only the webhook is).
- `lib/order/store.ts` — added `setOrderLsId(id, lsOrderId)` (status-agnostic patch; `lsOrderId` was in the contract but not writable post-create); patches **only** `ls_order_id` + `updated_at`, snake↔camel mapping stays single-source. `lib/catalog/products.ts` — `lsVariantId` documented as **env-resolved server-side at checkout** (never read in the client-safe catalog module — a deliberate divergence from the spec's "record in the catalog" wording, to keep the public storefront's static build pure); `priceUsd` stays the `$29` placeholder (the storefront **display** price — the real charge is configured on the LS variant; PM confirmed $29).
- `.env.local.example` — `LEMONSQUEEZY_API_KEY`, `LEMONSQUEEZY_STORE_ID`, `LEMONSQUEEZY_WEBHOOK_SECRET` (server-only secrets) + `LEMONSQUEEZY_VARIANT_STORY_1_BOOK` / `_STORY_2_LETTER` (non-secret runtime config). `lib/runtime/surface.boundary.test.ts` — both new routes added to `PUBLIC_API_ENTRIES` (verified engine-free; non-vacuous).
