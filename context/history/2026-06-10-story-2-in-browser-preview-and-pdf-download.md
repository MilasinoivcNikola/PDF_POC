Milestone 7 (Story 2), feature 19 — **Phase 5, completes Story 2.** Closed the Story-2 loop end to end (picker → wizard → generate → **preview → download**): after generation the "A Letter from [PET_NAME]" letter renders in-browser as **single-column** true-scale sheets with the real cover portrait + belief wash, the same letter components stream a parity-matched 6-page `Letter-from-[PET_NAME].pdf`, and the parent can inline-edit their own free-text/names or regenerate the cover. The render-pdf + regenerate routes were already registry-driven (features 14/16/17), so the work was a Story-2 editable-fields module, per-story dispatch through the registry, story-aware `/api/preview` + `/api/update-text`, and the single-column presentation. **Story-1 preview/download is behaviorally unchanged.**

- **Per-story dispatch folded into the registry.** `StoryDefinition` gained an `editable: EditableFieldsContract` field (string-typed at the boundary; the `StorySession | Story2Session` union narrowed per impl with a `storyType`-guarded cast, the same pattern `lib/ai/generate.ts` uses), so `getStory(session.storyType ?? "story-1").editable` is the single accessor `/api/preview`, `/api/update-text`, and the preview components share. `lib/story/story2/editable-fields.ts` (new, pure, tested) is the Story-2 page→field map — `letter-cover`: petName + ownerNames; `letter-page-3`: quirks + favoriteRitual; `letter-page-5`: favoriteSpots — all five required, `clean()` write-back into the right `Story2Session` groups, exhaustiveness-guarded. Story-1's `editable-fields.ts` gained a `FIELD_COPY` export (lifted verbatim from `PageView`) so both products own their editor copy in the same place.
- **`/api/preview` + `/api/update-text` story-aware.** Preview returns `storyType`, builds `fields` from the per-story contract, and never reads `.child` on a Story-2 session (`childName` only for Story 1). Update-text now **reads the session first** (the editable-field allowlist is per-story, so it needs `storyType`), then validates the field allowlist + required-blank **before any write**, **re-resolves via `getStory(...).resolve` before `writeSession`** (a `MergeError`→`story_incomplete` 422 leaves the on-disk letter untouched), and returns `childName: undefined` for Story 2. The `isSafeSessionId` traversal guard still runs before any read.
- **Preview presentation switches on `storyType`.** `BookPreview` keeps Story-1's facing-page `.spread` grid exactly as-is; Story 2 renders each letter page in a new screen-only `.preview-single` wrapper (centered, max-width capped — a letter reads as single sheets, not spreads). Both paths render the **shared feature-16 letter components** through `renderPage` (no markup fork), so the true-scale `--preview-scale = cellWidth/816` zoom and screen↔PDF parity both hold. Illustrated set + editable contract are derived from the registry (`getStory(storyType).illustrationSlots` / `.editable`); the feature-10 race guards (Download disabled while `downloading || regenerating || savingText`; `handleDownload` bails mid-flight; `downloadMeta` cleared after edit/regenerate) are intact for both stories. `PageView` is now generic (string-typed fields, injected `fieldCopy` + `isFieldRequired`) with Story-1 behavior byte-identical.
- **Client/server boundary fix (the one architectural call).** Importing the registry into the client preview pulled **Puppeteer into the client bundle** (registry → story defs → `render.ts`). Fixed by extracting the two pure filename builders into a new `lib/pdf/filename.ts` (the registry imports that; `render.ts` re-exports them for `scripts/render-test.ts` + `/api/render-pdf`), mirroring how feature 10 split `pages.tsx` out of `template.tsx`. The registry graph is now puppeteer-free; `/create/preview` still prerenders static. Story-1 filename byte-identical.
- **CSS — `app/globals.css` only.** The single-column wrapper is preview chrome with no print equivalent (same precedent as `.spread`, which lives only in `globals.css`). No shared `.story-page`/`.letter-page` selector or `--page-*` token was forked; tokens single-sourced via `var()`.
