## 2026-06-08 — In-Browser Preview & PDF Download

**Branch:** `feature/preview-and-download` → `main` (`446d960`, merge `10efc40`) · Craft Area 3 (nextjs-ui-builder) + 1 (PDF, support)

Milestone 5, feature 10 — **completes Milestone 5 and closes the end-to-end loop** (wizard 08 → generate 09 → **preview/download 10**). After generation the assembled book renders in the browser, page by page, with the real illustrations; the same template streams a print-quality PDF download; any single page can be repainted without rebuilding the book.

- **One template, two render targets (the central decision).** Extracted the per-page `.story-page` JSX out of feature-04's `lib/pdf/template.tsx` into a new **`lib/pdf/pages.tsx`** (`renderPage` / `StoryPages` — IO-free: no `node:`/fs, no `react-dom/server`). Both `renderStoryHtml` (PDF) and the screen `PageView` import it, so screen and PDF share one source of markup and **cannot structurally drift**. Verified **byte-identical** PDF output after the extraction (same length + SHA on the Otis fixture); print CSS untouched. To wire the API route → `render.ts` → `template.tsx` without tripping a Next.js 15 false-positive ("importing react-dom/server"), `template.tsx` now loads `react-dom/server` via a memoized synchronous `createRequire(import.meta.url)` require — `renderStoryHtml` stays synchronous and server-only; the client preview graph imports only the fs-free `pages.tsx`, so `/create/preview` still prerenders static with no fs/`react-dom/server` in the client bundle.
- `app/create/preview/page.tsx` + `components/preview/{BookPreview,PageView}.tsx` — the in-browser book. Client page reads the session id from `useWizard()` (`draft.id`, localStorage), fetches `GET /api/preview?id=` once, lays the resolved pages out as `prototypes/preview.html`'s facing-page spreads with the real generated watercolors. `PageView` renders the shared `renderPage` + a hover-revealed regenerate control on the 13 illustrated pages (back-cover excluded). No/invalid session degrades to a soft "couldn't find this book" notice — no white-screen.
- `app/api/render-pdf/route.ts` (was a 501 stub) — `POST { id }`: read session → `manifestToImageMap` → existing `renderStoryPdf` (feature 05, unchanged) → stream with `Content-Type: application/pdf`, `Content-Disposition: attachment; filename="Saying-Goodbye-to-[PET_NAME].pdf"` (via `storyPdfFilename`), and `Content-Length` so the UI shows real size.
- `app/api/preview/route.ts` (new) — `GET ?id=`: `resolveStory(session)` (the SAME resolved copy the PDF uses) + `manifestToImageMap` (data-URL images), returns `{ ok, pages, images, petName, childName }`. House JSON shape; non-`MergeError` resolve failures now return `{ ok:false, error:"preview_failed" }` (500) rather than a raw throw (review fix #2).
- `app/api/regenerate-illustration/route.ts` (new) — `POST { id, page }` → existing `regenerateSceneIllustration` (feature 07) → `spliceManifest` replaces exactly one manifest entry (no dupe) → `writeSession` → returns the new image as a `data:` URL the preview swaps in place. **No cache by design** (a regenerate is an explicit paid repaint of that one page); every other page stays cached/un-rebilled. Page gated to `SCENE_PAGE_IDS`; id traversal-guarded.
- All three new/updated routes guard `isSafeSessionId` before any disk access; `fs.readFile(entry.path)` reads only the server-derived path under `./generated/[id]`. Ported the preview screen styles (`.preview-header`, `.spread*`, regenerate control, `.download-final`, `.download-meta`, screen `.story-page*`) into `app/globals.css`, token-based.
- **Review fix #1:** the "Download PDF" buttons are now disabled while a repaint is in flight (`downloading || regenerating !== null`) and `handleDownload` bails if regenerating — so a mid-regenerate download can't capture the pre-repaint book.
