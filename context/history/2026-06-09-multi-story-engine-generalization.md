Milestone 7 (Story 2), feature 14 ‚Äî **Phase 0, the keystone refactor.** Generalized the render pipeline so the app can host a second product (Story 2, "A Letter from [PET_NAME]") **without changing any Story-1 output**. A behavior-preserving architecture change: the Story-1 PDF stays byte-identical and all existing tests stay green. Nothing user-facing changes ‚Äî this is pure scaffolding for features 15‚Äì19.

- **`storyType` discriminant (`lib/session/types.ts`)** ‚Äî new `StoryType = "story-1" | "story-2"` and an **optional** `storyType?: StoryType` on `StoryDraft`/`StorySession`. Left optional (not required) so legacy on-disk JSON deserializes cleanly; **zero migration** of `./sessions/*.json` ‚Äî every reader defaults a missing value via `?? "story-1"`. The `b41b8df0` fixture (no `storyType`) is the live proof.
- **`layout` dispatch (`lib/story/merge.ts` + `lib/pdf/pages.tsx`)** ‚Äî new `PageLayout` union (`cover ¬∑ dedication ¬∑ narrative ¬∑ truth ¬∑ love ¬∑ closing ¬∑ back-cover`) and a `layout` field on `ResolvedPage`, populated during Story-1 merge by a `STORY_1_LAYOUT` map that is **one-for-one with the old per-id treatment** (cover‚Üícover, page-1‚Üídedication, page-7‚Üítruth, page-10‚Üílove, page-12‚Üíclosing, back-cover‚Üíback-cover, the other 8 narrative pages‚Üínarrative). `renderPage` now `switch`es on `page.layout` with **no literal Story-1 page-id remaining** ‚Äî exhaustive over `PageLayout`, no `default`, so a future Story-2 layout that forgets a case is a compile-time failure. This is the seam that stops a letter page from silently falling through to the children's-book `NarrativePage`. **Page ids stay stable**, so manifest/cache/regenerate keys are untouched.
- **Story registry (`lib/story/registry.ts` + `lib/story/story-1.ts`, new)** ‚Äî `StoryDefinition { resolve(session); illustrationSlots; pdfFilename(session) }` (room for `wizardSteps` later) + `getStory(storyType)`. Story-1 registered by **wrapping** the existing `resolveStory` / `SCENE_PAGE_IDS` / `storyPdfFilename` with no logic change; `illustrationSlots` is the same set **and order** as `SCENE_PAGE_IDS`. The `"story-2"` entry is deliberately absent (`getStory("story-2")` throws a clear message; unreachable for current sessions until feature 15).
- **Route wiring (`lib/pdf/render.ts` + `app/api/{render-pdf,preview,generate-illustrations,regenerate-illustration}/route.ts`)** ‚Äî all resolve story / filename / slots via `getStory(session.storyType ?? "story-1")` instead of importing Story-1 modules directly. `regenerate-illustration` keeps its scene-slot allowlist validation **before** the session read (uses `getStory("story-1").illustrationSlots` directly ‚Äî storyType isn't known pre-read, Story 1 is the only product), so the existing `invalid_request` error contract and feature-09's synchronous TOCTOU idempotency guard are intact. House JSON shape preserved across all four routes.
- **Import cycle** (`render.ts ‚Üí registry.ts ‚Üí story-1.ts ‚Üí render.ts`) is safe: `story-1.ts` eagerly reads only `SCENE_PAGE_IDS` (from `prompts.ts`, outside the cycle), and `resolveStory`/`storyPdfFilename` are referenced lazily inside the definition's methods via ESM live bindings ‚Äî no module-eval-time `getStory()` call. `resolveStory`'s own export/signature is unchanged, so Story-1 callers and tests needed no rewrites.
