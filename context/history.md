## 2026-06-09 — Input Photo Downscale

**Branch:** `feature/input-photo-downscale` → `main` (`10c3873`, merge `56f5136`) · Craft Area 2 (AI illustration), implemented in Craft Area 3 (wizard uploader)

Milestone 6, polish item 12 (from `12-input-photo-downscale.md`). The uploaded pet photo is re-sent to OpenAI as a reference on **all 14** illustration calls per book (1 reference + 13 scenes); a 2400×1600 phone photo is far more than `gpt-image-2` needs to anchor the pet. Cap the photo's longest edge at ≤1024px **once**, before upload, so every downstream `images.edit` call ships a smaller reference → fewer input-image tokens, lower cost, faster uploads. The "free" optimization — output resolution and the printed PNGs are untouched.

- **Approach decision — Option 1 (client `<canvas>`), chosen over server/`sharp` options.** The repo has no image-processing lib and `coding-standards.md` forbids new deps without sign-off. Three viable downscale sites were surfaced: (1) client canvas in the uploader (**zero deps**, also shrinks the upload), (2) server-on-upload (+`sharp`), (3) at generation in `lib/ai/` (+`sharp`, preserves the full-res original on disk). Picked **Option 1** — no dependency, cuts upload bandwidth, and the "lose the pristine original" concern is moot while we're not using `input_fidelity: "high"` (out of scope). Routed to `nextjs-ui-builder` (not the spec's default `ai-image-specialist`) because the chosen option's code lives in `components/wizard/ImageUploader.tsx`, their territory.
- `components/wizard/downscaleImage.ts` (new) — the resize helper. **`targetDimensions(source, maxEdge=1024)`** is the pure, DOM-free unit-test seam: caps the longest edge, preserves aspect via a single scale factor, **never upscales** (≤1024 returns dims unchanged — exact-1024 passes through), floors the short edge to ≥1px so a panorama can't collapse to 0. **`downscaleImage(file, maxEdge?)`** does the canvas work — `createImageBitmap(file, { imageOrientation: "from-image" })` (bakes EXIF rotation into upright pixels so phone photos don't upload sideways) → `<canvas>.drawImage` → `toBlob("image/jpeg", 0.85)` → a `.jpg` `File`; already-within-cap returns the **original** `File` untouched (no re-encode, no quality loss). `downscaledName` swaps the extension. Mirrors the `extensionToMime` (pure, tested) vs `imageToDataUrl` (IO, manual) split.
- `components/wizard/ImageUploader.tsx` — wires the downscale **after** the existing MIME (jpeg/png/webp) + dimension validation and **before** the POST to `/api/upload`; on any decode/encode failure it falls back to uploading the original file rather than blocking; `bitmap.close()` in a `finally`. Logs the byte delta (`[downscale] <name>: <orig> → <new> bytes`, `(unchanged — already within cap)` on passthrough) for QA evidence. The preview/low-res-notice/replace behavior is unchanged (still describes the user's chosen original).
- **Determinism is load-bearing.** The AI cache (`lib/ai/cache.ts` → `hashReferenceSet`) keys on the photo bytes. Fixed `MAX_EDGE=1024` + `JPEG_QUALITY=0.85` consts, integer-rounded dims, no `Date`/`Math.random` → the same photo yields the same bytes, so the **one-time cache-key shift** (smaller bytes → new hash → one miss → regenerate, then stable) is expected behavior, not a bug. Tests assert both consts as the determinism guard.
- **No new dependency** (`package.json`/lock untouched) — the whole point of Option 1.

**Live QA (PASS, $0).** Drove the real wizard in a browser: 2400×1600 JPEG → console `309900 → 61525 bytes` (~80% smaller), on disk **1024×683**, aspect preserved (1.4993 vs 1.5000); portrait 1600×2400 → 683×1024. 800×600 photo → `(unchanged — already within cap)`, saved file **md5 byte-identical** to original. Validation not weakened — `.txt` rejected client-side with no `[downscale]` log and no POST; server still returns 415/413/400 for bad type / >10 MB / empty. EXIF Orientation=6 image uploaded **upright** (683×1024, would've been 1024×683 sideways if ignored). No image generation ran.

**Known limitation carried forward (verify on next live run):** the **pet-consistency** goal — "a 1024px reference still reads as the same animal vs. the full-size photo" — was **not** empirically verified. The `.env.local` `OPENAI_API_KEY` is invalid (free `GET /v1/models` → 401), so no paid Low-tier comparison run was possible; also the documented `b41b8df0` baseline came from a 640×640 photo (under the cap → downscale no-ops), so a fair comparison needs a genuine high-res source. Risk judged low on the merits (gpt-image-2 generates at 1024×1024, so a 1024px-long-edge reference carries at least as much detail as the output) — but a deliberate one-shot Low run with a valid key + high-res photo remains the honest confirmation. Per spec: keep it **Low, once**.

**Gates:** `npm run build` ✓ (22 routes, `/create/upload` still prerenders static) · `npm run test:run` ✓ (**370 tests, +16** — `components/wizard/downscaleImage.test.ts`: `targetDimensions` no-upscale/landscape/portrait/square/cap-never-exceeded/panorama-floor/custom-maxEdge, `downscaledName` cases, the const guard) · `qa` **PASS** (3/3 core + EXIF, $0; consistency blocked on key) · code review **PASS** (no blockers; 3 cosmetic nice-to-haves deferred: prod-gate the console.logs, preview card shows pre-downscale dims/size, EXIF-axis asymmetry in the low-res notice — all rotation-invariant for the cap, so correctness holds).
