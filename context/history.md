**Branch:** `feature/input-photo-downscale` ŌåÆ `main` (`10c3873`, merge `56f5136`) ┬Ę Craft Area 2 (AI illustration), implemented in Craft Area 3 (wizard uploader)

- **Approach decision ŌĆö Option 1 (client `<canvas>`), chosen over server/`sharp` options.** The repo has no image-processing lib and `coding-standards.md` forbids new deps without sign-off. Three viable downscale sites were surfaced: (1) client canvas in the uploader (**zero deps**, also shrinks the upload), (2) server-on-upload (+`sharp`), (3) at generation in `lib/ai/` (+`sharp`, preserves the full-res original on disk). Picked **Option 1** ŌĆö no dependency, cuts upload bandwidth, and the "lose the pristine original" concern is moot while we're not using `input_fidelity: "high"` (out of scope). Routed to `nextjs-ui-builder` (not the spec's default `ai-image-specialist`) because the chosen option's code lives in `components/wizard/ImageUploader.tsx`, their territory.
- `components/wizard/downscaleImage.ts` (new) ŌĆö the resize helper. **`targetDimensions(source, maxEdge=1024)`** is the pure, DOM-free unit-test seam: caps the longest edge, preserves aspect via a single scale factor, **never upscales** (Ōēż1024 returns dims unchanged ŌĆö exact-1024 passes through), floors the short edge to Ōēź1px so a panorama can't collapse to 0. **`downscaleImage(file, maxEdge?)`** does the canvas work ŌĆö `createImageBitmap(file, { imageOrientation: "from-image" })` (bakes EXIF rotation into upright pixels so phone photos don't upload sideways) ŌåÆ `<canvas>.drawImage` ŌåÆ `toBlob("image/jpeg", 0.85)` ŌåÆ a `.jpg` `File`; already-within-cap returns the **original** `File` untouched (no re-encode, no quality loss). `downscaledName` swaps the extension. Mirrors the `extensionToMime` (pure, tested) vs `imageToDataUrl` (IO, manual) split.
- `components/wizard/ImageUploader.tsx` ŌĆö wires the downscale **after** the existing MIME (jpeg/png/webp) + dimension validation and **before** the POST to `/api/upload`; on any decode/encode failure it falls back to uploading the original file rather than blocking; `bitmap.close()` in a `finally`. Logs the byte delta (`[downscale] <name>: <orig> ŌåÆ <new> bytes`, `(unchanged ŌĆö already within cap)` on passthrough) for QA evidence. The preview/low-res-notice/replace behavior is unchanged (still describes the user's chosen original).
- **Determinism is load-bearing.** The AI cache (`lib/ai/cache.ts` ŌåÆ `hashReferenceSet`) keys on the photo bytes. Fixed `MAX_EDGE=1024` + `JPEG_QUALITY=0.85` consts, integer-rounded dims, no `Date`/`Math.random` ŌåÆ the same photo yields the same bytes, so the **one-time cache-key shift** (smaller bytes ŌåÆ new hash ŌåÆ one miss ŌåÆ regenerate, then stable) is expected behavior, not a bug. Tests assert both consts as the determinism guard.
- **No new dependency** (`package.json`/lock untouched) ŌĆö the whole point of Option 1.
